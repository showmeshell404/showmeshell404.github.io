Androguard

2018-08-19

Androguard的使用方法

  • androapkinfo.py

    • 用来查看apk文件的信息
    • 会输入apk文件的包、资源、权限、组件、方法等信息，输出的内容比较详细，可将输出信息重定向到文件后再进行查看
    • ./androapkinfo.py -i ./xxx.apk

  • androaxml.py

    • 用来解密apk包中的AndroidManifest.xml文件
    • ./androaxml.py -i ./xxx.apk

  • androcsign.py

    • 用于添加apk文件的签名信息到一个数据库文件中
    • Androguard工具目录下的signatures/dbandroguard文件为收集的恶意软件信息数据库
    • 使用androcsign.py前需要为apk文件编写一个sign文件,json格式：eg

    (json不允许/* //注释，需要则去掉注释部分)

    • ./androcsign.py -i signatures/xxx.apk -o signatures/dbandroguard

  • androdd.py

    • 用来生成apk文件中每个类的方法的调用流程图
    • ./androdd.py -i ./xxx.apk -o ./out -d -f PNG
      • -o选项强制指定一个输入目录，-d指定生成dot图形文件，-f用来指定输出的图片格式

  • androdiff.py

    • 用来比较两个apk文件的差异
    • ./androdiff.py -i ./xxx.apk ./xxx1.apk

  • androdump.py

    • 用来dump一个Linux进程的信息
    • ./androdump.py -i pid

  • androgexf.py

    • 用来生成APK的GEXF格式的图形文件
    • 可以使用Gephi查看gexf格式的图像
    • ./androgext.py -i ./xxx.apk -o ./xxx.gexf

  • andromercury.py

    • Mercury工具的框架
    • 功能是对Mercury的包装

  • androrisk.py

    • 用于评估apk文件中潜在的风险
    • ./androrisk.py -m -i ./xxx.apk

  • androsign.py

    • 用于检测apk的信息是否存在于特定的数据库中，作用与androcsign.py相反
    • ./androsign.py -i apks/xxx.apk -b signatures/dbandroguard -c signatures/dbconfig

  • androsim.py

    • 用于计算两个apk文件的相似度
    • ./androsim.py -i ./xxx.apk ./xxx1.apk
    • Windows移植版=androsim.exe

  • androxgmml.py

    • 用来生成apk/jar/class/dex文件的控制流程及功能调用图，输出格式为xgmml
      ./androxgmml.py -i ./xxx.apk -o xxx.xgmml

  • apkviewer.py

    • 用来为apk文件中每一个类生成一个独立的graphml图形文件
    • ./apkviewer.py -i ./xxx.apk -o output
    • Gephi打开生成后的graphml文件,图形中的每个节点是指令级别

  • androlyze.py

    • 强大的静态分析工具，它提供的一个独立的Shell环境
    • ./androlyze.py-s进入androlyze的shell交互环境
    • androlyze.py通过访问对象的字段与方法的方式来提供反馈结果，可用到3个对象：apk文件对象、dex文件对象、分析结果对象
      • apk文件对象 a=APK("./xxx.apk") (a.+tab 可显示该类所有方法与字段)
      • dex文件对象d=DalvikVMFormat(a.get_dex())
      • 分析结果对象 dx=VMAnalysis(d)
      • 或一条命令解决 a,d,dx=AnalyzeAPK("./xxx.apk", decompiler="dad")

dex2jar用法

2018-08-19
  • dex2jar用法
    • d2j-apk-sign用来为apk文件签名。命令格式：d2j-apk-sign xxx.apk。
    • d2j-asm-verify用来验证jar文件。命令格式：d2j-asm-verify -d xxx.jar。
    • d2j-dex2jar用来将dex文件转换成jar文件。命令格式：d2j-dex2jar xxx.apk
    • d2j-dex-asmifier用来验证dex文件。命令格式：d2j-dex-asmifier xxx.dex。
    • d2j-dex-dump用来转存dex文件的信息。命令格式：d2j-dex-dump xxx.apk out.jar。
    • d2j-init-deobf用来生成反混淆jar文件时的初始化配置文件。
    • d2j-jar2dex用来将jar文件转换成dex文件。命令格式：d2j-jar2dex xxx.apk。
    • d2j-jar2jasmin用来将jar文件转换成jasmin格式的文件。命令格式：d2j-jar2jasmin xxx.jar
    • d2j-jar-access用来修改jar文件中的类、方法以及字段的访问权限。
    • d2j-jar-remap用来重命名jar文件中的包、类、方法以及字段的名称。
    • d2j-jasmin2jar用来将jasmin格式的文件转换成jar文件。命令格式：d2j-jasmin2jar dir
    • dex2jar为d2j-dex2jar的副本。
    • dex-dump为d2j-dex-dump的副本

Java代码结构与Smali代码语句分析

1.循环语句

Android开发过程中，常见的循环结构有迭代器循环、for循环、while循环、do while循环。

  • 迭代器循环代码：

  • 迭代器循环特点：

    • 迭代器循环会调用迭代器的hasNext()方法检测循环条件是否满足
    • 迭代器循环中调用迭代器的next()方法获取单个对象
    • 循环中使用goto指令来控制代码的流程
    • for形式的迭代器循环展开后即为while形式的迭代器循环

  • for循环

  • for循环的特点

    • 在进入循环前，需要先初始化循环计数器变量，且它的值需要在循环体中更改
    • 循环条件判断可以是条件跳转指令组成的合法语句
    • 循环中使用goto指令来控制代码的流程

2.switch语句

1.有规律的case分支

代码中的switch分支使用的是packed-switch指令。packed-switch指令在Dalvik中的格式如下：packed-switch vAA, +BBBBBBBB

指令后面的“+BBBBBBBB”被指明为一个packed-switch-payload格式的偏移:格式如下：

IDA Pro找到“packed-switch p1, :pswitch_data_0”指令位于0x2cb1a处，相应的机器码为“2B 02 13 00 00 00”:2B为packed-switch的OpCode,02为寄存器p1,00000013为偏移量0x13。Dalvik中计算偏移是以两个字节为单位，因为实际该指令指向的packed-switch-payload结构体的偏移量为0x2cb1a + 2 * 0x13 = 0x2cb40

  • 第1个ident字段为0x100，标识packed-switch有效的case区域。
  • 第2个字段size为4，表明有4个case。
  • 第3个字段first_key为0，表明初始case值为0。
  • 第4个字段为偏移量，分别为0x6、0x9、0xc、0xf，加上packed-switch指令的偏移值0x2cb1a
    • case 0位置 = 0x2cb1a + 2 * 0x6 = 0x2cb26
    • case 1位置 = 0x2cb1a + 2 * 0x9 = 0x2cb2c
    • case 2位置 = 0x2cb1a + 2 * 0xc = 0x2cb32
    • case 3位置 =0x2cb1a + 2 * 0xf = 0x2cb38
2.无规律的case分支

sparse-switch指令在Dalvik中的格式如下：sparse-switch vAA, +BBBBBBBB

指令后面的“+BBBBBBBB”被指明为一个sparse-switch-payload格式的偏移。它的格式如下

IDA Pro找到“sparse-switch p1, :sswitch_data_0”指令位于0x2cb6a处，相应的机器码为“2C 02 13 00 00 00”，2C为sparse -switch的OpCode,02为寄存器p1,00000013为偏移量0x13。因为实际该指令指向的sparse-switch-payload结构体的偏移量为0x2cb6a + 2 * 0x13 = 0x2cb90

  • 第1个ident字段为0x200,标识sparse-switch有效的case区域
  • 第2个字段size为4，表明有4个case。
  • 第3个字段keys为4个case的值，分别为0x5、0xf、0x23、0x41。
  • 第4个字段为偏移量，分别为0x6、0x9、0xc、0xf，加上sparse -switch偏移值0x2cb6a
    • case 0位置 =0x2cb6a + 2 * 0x6 = 0x2cb76
    • case 1位置 = 0x2cb6a + 2 * 0x9 = 0x2cb7c
    • case 2位置 = 0x2cb6a + 2 * 0xc = 0x2cb82
    • case 3位置 = 0x2cb6a + 2 * 0xf = 0x2cb88

3.try/catch语句

try语句块使用try_start_开头的标号注明，以try_end_开头的标号结束。第一个try语句的开头标号为try_start_0，结束标号为try_end_0

try_end_0标号下面使用“.catch”指令指定处理到的异常类型与catch的标号，格式如下

.catch <异常类型> {\<try起始标号> .. \<try结束标号>} \<catch标号>

Android中的静态分析

2018-08-18

静态分析

  • 如何静态分析Android程序：

    • 阅读反汇编生成的Dalvik字节码，可以使用IDA Pro分析dex文件
    • 使用文本编辑器阅读baksmali反编译生成的smali文件
    • 阅读反汇编生成的Java源码，可以使用dex2jar生成jar文件，然后再使用jd-gui阅读jar文件的代码

  • 如何快速定位Android程序的关键代码

    • AndroidManifest.xml文件
      • 记录着软件的一些基本信息，包括软件的包名、运行的系统版本、用到的组件等。

    • 程序的主Activity

      • 每个Android程序有且只有一个主Activity（隐藏程序除外，它没有主Activity），它是程序启动的第一个Activity

      • android:label指定Activity的标题

      • android:name指定具体的Activity类
        • “.MainActivity”前面省略了程序的包名，完整类名应该为com.droider.xxx.MainActivity
      • intent-filter指定了Activity的启动意图
        • android.intent.action.MAIN表示这个Activity是程序的主Activity
        • android.intent.category.LAUNCHER表示这个Activity可以通过LAUNCHER来启动
        • (如果AndroidMenifest.xml中，所有的Activity都没有添加android.intent.category.LAUNCHER，那么该程序安装到Android设备上后，在程序列表中是不可见的,如果没有指定android.intent.action.MAIN，Android系统的LAUNCHER就无法匹配程序的主Activity，因此该程序也不会有图标出现)
      • 找到主Activity后，可以直接去查看其所在类的OnCreate()方法的反汇编代码:大多数程序的代码入口处。
    • Application类
      • 在程序的组件之间传递全局变量，或在Activity启动之前做初始化工作
      • 类继承自android.app.Application，重写它的OnCreate()方法

  • 如何定位关键代码

    • 信息反馈
      • 字符串做为突破点：2种
        • 字符串会存储在String.xml文件：
          • 以id的形式访问，在反汇编代码中搜索字符串的id值
        • 字符串硬编码到程序代码:
          • 反汇编代码中直接搜索字符串
    • 特征函数法
      • 反馈需调用Android SDK中提供的相关API函数来完成
        • eg:弹出注册码错误的提示信息需调用Toast.MakeText().Show()方法，在反汇编代码中搜索Toast就可定位
    • 顺序查看法
      • 软件的启动代码开始，逐行的向下分析，执行流程
    • 代码注入法
      • 手动修改apk文件的反汇编代码，加入Log输出，配合LogCat查看程序执行到特定点时的状态数据
    • 栈跟踪法
      • 输出运行时的栈跟踪信息，然后查看栈上的函数调用序列来理解方法的执行流程
    • Method Profiling
      • 记录每个函数占用的CPU时间外，还可跟踪所有的函数调用关系，并提供比栈跟踪法更详细的函数调用序列报告

Android程序中的类

2018-08-18

Android程序中的类

分析Android程序中的类，便于分析反编译后文件的层次结构与文件名对于的含义。

1.内部类（Inner Class）

类中定义的类为内部类（Inner Class），可分为成员内部类、静态嵌套类、方法内部类、匿名内部类

内部类作为一个独立的类，拥有自己独立的smali文件

  • 内部类的文件名形式为[外部类]$[内部类].smali

  • 观察实例字段：sn是字符串类型，this$0是MainActivity类型，synthetic关键字表明它是“合成”的

    • this$0是内部类自动保留的一个指向所在外部类的引用
    • this表示为父类的引用，数值0表示引用的层数

  • 构造函数，初始化过程：

使用“.parameter”指令指定了两个参数，而实际上却使用了p0～p2共3个寄存器：

对于一个非静态的方法而言，会隐含的使用p0寄存器当作类的this引用，p0表示MainActivity$SNChecker自身的引用，p1表示MainActivity的引用，p2表示sn字符串

  • 内部类的初始化步骤：
    • 保存外部类的引用到本类的一个synthetic字段中，以便内部类的其它方法使用
    • 然后是调用内部类的父类的构造函数来初始化父类
    • 最后是对内部类自身进行初始化

2.监听器

Android开发中使用了大量监听器，比如Button的点击事件响应OnClickListener、Button的长按事件响应OnLongClickListener、ListView列表项的点击事件响应OnItemSelectedListener等。

实际编写代码的过程中，多采用匿名内部类的形式来实现

按钮传入了一个MainActivity$1对象的引用,在MainActivity$1.smali文件的开头使用了“.implements”指令指定该类实现了按钮点击事件的监听器接口。

因此：这个类实现了它的OnClick()方法。

3.注解类

  • Android系统中涉及到注解的包共有两个
    • 一个是dalvik.annotation
      • 该程序包下的注解不对外开放，仅供核心库与代码测试使用
      • 注解声明位于Android系统源码的libcore\dalvik\src\main\java\dalvik\annotation目录下
    • 一个是android.annotation
      • 注解声明位于Android系统源码的frameworks\base\core\java\android\annotation目录下
  • 注解类大概有以下几种
  • MemberClasses

MemberClasses注解是一个“系统注解”，作用是为父类提供一个MemberClasses列表。即子类成员集合，一个内部类列表

  • EnclosingMethod

EnclosingMethod注解用来说明整个类的作用范围,Method表明它作用于一个方法,value表明它位于MainActivity的onCreate()方法中

  • EnclosingClass

EnclosingClass注解表明其作用于一个类，value表明这个类是MainActivity

  • InnerClass

InnerClass表明自身是一个内部类

  • AnnotationDefault

注解类在声明时提供了默认值，那么程序中会使用到AnnotationDefault注解。

  • Signature

Signature注解用于验证方法的签名,在smali中方法的原型与Signature注解的value值是一致的。

  • Throws

如果方法的声明中使用throws关键字抛出异常，则会生成相应的Throws注解。

以上的注解都是自动生成的，用户不可以在代码中添加使用

  • SuppressLint

作用是辅助开发人员去除代码检查器（Lint API check）添加的警告信息

  • TargetApi

程序代码中使用到的API等级比AndroidManifest.xml文件中定义的minSdkVersion要高，则在方法或方法所在类的前面添加“@TargetApi(9)”

4.自动生成的类

  • R类

    • 工程res目录下的每个资源都会有一个id值，这些资源的类型可以是字符串、图片、样式、颜色
    • 这些资源类都是R类的内部类，因此它们都会独立生成一个类文件
    • R.smali、R$attr.smali、R$dimen.smali、R$drawable.smali、R$id.smali等

  • BuildConfig类

    • 类中只有一个boolean类型的名为DEBUG的字段
    • 用来标识程序发布的版本类型
    • 值默认是true，即程序以调试版本发布

  • 注解类

    • 代码中使用了SuppressLint或TargetApi注解
    • smali\android\annotation目录下生成相应的smali文件

  • support

    • 提供了高版本才有的如Fragment、ViewPager等控件供低版本的程序调用情况
    • smali\android\support目录下生成相应的smali文件

smali文件格式

普通类、抽象类、接口类或者内部类，在反编译出的代码中，都以单独的smali文件来存放

1.smali文件头3行

经过混淆的dex文件，反编译出来的smali代码可能没有源文件信息，“.source”行的代码可能为空。

前3行代码过后就是类的主体部分了，一个类可以由多个字段或方法组成

2.smali字段

smali文件中字段的声明使用“.field”指令

字段有静态字段与实例字段两种

  • 静态字段的声明
    • # static fields
    • .field <访问权限> static [修饰关键字] <字段名>:<字段类型>
  • 实例字段的声明
    • # instance fields
    • .field <访问权限> [修饰关键字] <字段名>:<字段类型>

3.smali方法

smali文件中方法的声明使用“.method”指令

方法有直接方法与虚方法两种

  • 直接方法的声明

  • 虚方法的声明与直接方法相同，只是起始处的注释为“virtual methods”

4.smali接口

smali文件中使用“.implements”指令指出一个类实现了接口

  • 格式声明

    • # interfaces
    • .implements <接口名>
      • 接口名是DexClassDef结构中interfacesOff字段指定的内容

5.smali注解

smali文件中使用“.annotation”指令指出类使用了注解

  • 格式声明

    • 注解的作用范围可以是类、方法或字段

      • 注解的作用范围是类，“.annotation”指令会直接定义在smali文件中
      • 注解的作用范围是方法或字段，“.annotation”指令则会包含在方法或字段定义中

dex文件验证

2018-08-17

dex文件的验证

  • 验证与优化dex文件最简单且最安全的方法是直接在虚拟机中加载dex文件
    • 一旦程序加载失败，就说明dex文件未优化或验证失败，此时中止程序运行即可
      • 但会引起部分资源难以从内存中释放，比如加载的native动态库
      • 让验证优化工作与需要执行的代码在同一虚拟机中运行不是很好
  • Android提供了一个专门验证与优化dex文件的工具dexopt
    • 在加载一个dex文件时，通过指定的验证与优化选项来调用dexopt进行相应的验证与优化操作。

dexopt的整个验证与优化过程：

  • dexopt的主程序代码为OptMain.cpp
  • 处理apk/jar/zip文件中classes.dex的函数为extractAndProcessZip()
  • 通过dexZipFindEntry()函数检查目标文件中是否拥有classes.dex
  • 调用dexZipGetEntryInfo()函数来读取classes.dex的时间戳与crc检验值
  • 调用dexZipExtractEntryTo-File()函数释放classes.dex为缓存文件
    • 开始解析传递过来的验证与优化选项，验证选项使用“v=”指出，优化选项使用“o=”指出
  • 调用dvmPrepForDexOpt()函数启动一个虚拟机进程
    • 优化选项dexOptMode与验证选项verifyMode被传递到了全局DvmGlobals结构gDvm的dexOptMode与classVerifyMode字段中,初始化工作完成
  • dexopt调用dvmContinueOptimization()函数开始真正的验证与优化工作
    • 函数首先对dex文件做简单的检查，确保传递进来的目标文件属于dex或odex
    • 接着调用mmap()函数将整个文件映射到内存中
    • 然后根据gDvm的dexOptMode与classVerifyMode字段来设置doVerify与doOpt两个布尔值
    • 接着调用rewriteDex()函数来重写dex文件
      • 重写内容包括字符序调整、结构重新对齐、类验证信息以及辅助数据
  • rewriteDex()函数调用dexSwapAndVerify()调整字节序
  • 调用dvmDexFileOpenPartial()创建DexFile结构
    • dvmDexFileOpenPartial()函数的实现在Android系统源码dalvik\vm\DvmDex.cpp文件
    • 该函数调用dexFileParse()函数解析dex文件
      • dexFileParse()函数读取dex文件的头部
      • 根据需要调用验证dexComputeChecksum()函数或调用dexComputeOptChecksum()函数来
      • 验证dex或odex文件头的checksum与signature字段
      • 验证成功后
    • 调用allocateAuxStructures()函数设置DexFile结构辅助数据的相关字段
    • 执行完后返回到rewriteDex()函数
  • rewriteDex()调用loadAllClasses()加载dex文件中所有的类
  • 调用verifyAndOptimizeClasses()函数优化与验证具体的类
    • 调用dvmVerifyClass()函数进行验证
      • 函数的实现代码位于Android系统源码dalvik\vm\analysis\DexVerify.cpp文件
      • 调用verifyMethod()函数对类的所有直接方法与虚方法进行验证
        • 先调用verifyInstructions()函数来验证方法中的指令及其数目的正确性
        • 再调用dvmVerifyCodeFlow()函数来验证代码流的正确性
    • 调用dvmOptimizeClass()函数进行优化
      • 函数的实现代码位于Android系统源码dalvik\vm\analysis\Optimize.cpp文件
      • 调用optimizeMethod()函数对类的所有直接方法与虚方法进行优化
        • 优化的主要工作是进行“指令替换”
          • 替换原则的优先级为“volatile”替换-正确性替换-高性能替换
    • rewriteDex()函数返回后
  • 再次调用dvmDexFileOpenPartial()来验证odex文件
  • 调用dvmGenerateRegisterMaps()函数来填充辅助数据区结构
  • 调用updateChecksum()函数重写dex文件的checksum值

完整的流程，学习的一遍，对于dex如何优化，大概流程已经知道啦~

odex文件格式

odex是OptimizedDEX的缩写，表示经过优化的dex文件。

1.生成odex文件

  • odex存在方式

    • 一种是从apk程序中提取出来，与apk文件存放在同一目录且文件后缀为odex的文件，这类odex文件多是Android ROM的系统程序
    • 一种是dalvik-cache缓存文件，这类odex文件仍然以dex作为后缀，存放在cache/dalvik-cache目录下
      • 保存的形式为“apk路径@apk名@classes.dex”，eg：“system@app@Calculator.apk@classes.dex”表示安装在/system/app目录下Calculator.apk程序的odex文件。

  • odex文件的优势

    • 由于Android程序的apk文件为zip压缩包格式，Dalvik虚拟机每次加载它们时需要从apk中读取classes.dex文件，这样会耗费很多cpu时间。
    • 采用odex方式优化的dex文件，已经包含了加载dex必须的依赖库文件列表，Dalvik虚拟机只需检测并加载所需的依赖库即可执行相应的dex文件，这大大缩短了读取dex文件所需的时间。
    • 对于部分Android系统的ROM，由于将系统app全部转换成外置的odex文件与apk放在同一目录，这样系统在启动加载这些程序时会节省更多的时间，启动速度自然也会更快。

  • dexopt-wrapper

    • 生成odex文件的工具dexopt-wrapper

    • dexopt-wrapper主程序源码DexOptWrapper.cpp文件，实现调用的是/system/bin/dexopt程序

odex文件在dex文件头部添加了一些数据，然后在dex文件尾部添加了dex文件的依赖库以及一些辅助数据

  • Dalvik虚拟机将dex文件映射到内存中后是DexFile格式

    • DexLink以下的部分被称为“auxillary section”，即辅助数据段，它记录了dex文件被优化后添加的一些信息
    • DexFile结构描述的是加载进内存的数据结构，还有一些数据是不会加载进内存的

dex文件格式

Android 4.0源码Dalvik/docs目录下文档dex-format.html有详细介绍dex文件格式

1.dex文件中的数据结构

dex文件使用到的数据类型

  • u1～u8表示1到8字节的无符号数，而sleb128、uleb128与uleb128p1则是dex文件中特有的LEB128数据类型。
  • 每个LEB128由1～5个字节组成，所有的字节组合在一起表示一个32位的数据。
  • LEB128，每个字节只有7位为有效位，如果第1个字节的最高位为1，表示LEB128需要使用到第2个字节，如果第2个字节的最高位为1，表示会使用到第3个字节，以此类推，直到最后的字节最高位为0，当然，LEB128最多只会使用到5个字节，如果读取5个字节后下一个字节最高位仍为1，则表示该dex文件无效，Dalvik虚拟机在验证dex时会失败返回。

对于有符号的LEB128（sleb128）来说，计算方法与无符号的LEB128是一样的，只是对无符号LEB128最后一个字节的最高有效位进行了符号扩展。读取有符号LEB128数据的代码如下：

uleb128p1类型很简单，值为uleb128的值加1。

c0 83 92 25计算uleb128值：

第1个字节0xc0大于0x7f，表示需要用到第2个字节。result=0xc0&0x7f

第2个字节0x83大于0x7f，表示需要用到第3个字节。result2 = result1 + (0x83 & 0x7f) << 7

第3个字节0x92大于0x7f，表示需要用到第4个字节。result3 = result2 + (0x92 & 0x7f) << 14

第4个字节0x25小于0x7f，表示到了结尾。result4 = result3 + (0x25 & 0x7f) << 21

最后计算结果为0x40 + 0x180 + 0x48000 + 0x4a00000 = 0x4a481c0

2.dex文件整体结构

dex文件由多个结构体组合而成。

  • dex header：dex文件头，它指定了dex文件的一些属性，并记录了其它6部分数据结构在dex文件中的物理偏移
  • string_ids到class_def结构为“索引结构区”，真实的数据存放在data数据区
  • link_data：静态链接数据区

未经过优化的dex文件结构：

(DexFile结构的声明在Android系统源码dalvik\libdex\DexFile.h文件中的，是dex文件被映射到内存中的结构，保存的是各个结构的指针，其中还包括了DexOptHeader与DexFile尾部的附加数据)

  • DexHeader结构占用0x70个字节

  • 70个字节解释如下：
    • magic字段标识了一个有效的dex文件，值固定为64 65 78 0a 30 33 35 00，转换为字符串为dex.035
    • checksum段为dex文件的校验和，通过它来判断dex文件是否被损坏或篡改。
    • signature字段用来识别最佳化之前的dex文件
    • fileSize字段记录了包括DexHeader在内的整个dex文件的大小
    • headerSize字段记录了DexHeader结构本身占用的字节数，目前它的值为0x70。
    • endianTag字段指定了dex运行环境的cpu字节序，预设值ENDIAN_CONSTANT等于0x12345678，表示默认采用Little-Endian字节序
    • linkSize字段与linkOff字段指定链接段的大小与文件偏移，大多数情况下它们的值都为0。
    • mapOff字段指定了DexMapList结构的文件偏移
    • 接下来的字段分别表示DexStringId、DexTypeId、DexProtoId、DexFieldId、DexMethodId、DexClassDef以及数据段的大小与文件偏移。
      • DexHeader结构下面的数据为”索引结构区“与”数据区“
  • “索引结构区”中各数据结构的偏移地址都是从DexHeader结构的stringIdsOff～classDefsOff字段的值指定的，它们并非真正的类数据，而是指向dex文件的data数据区（DexData字段，实际上是ubyte字节数组，包含了程序所有使用到的数据）的偏移或数据结构索引。
